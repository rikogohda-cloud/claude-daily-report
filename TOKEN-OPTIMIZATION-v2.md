# トークン最適化設計 v2（質を落とさない版）

## 目標
月間トークン消費量を **3〜4.5％**（60〜90万トークン）に抑える。
**データの網羅性・質は維持する。**

## 現状分析（2026-02-25 実測値）
- 1回あたりのトークン消費: 約90,000トークン
- 月30回実行: 約270万トークン（**13.5％**）

## 最適化戦略（データ収集は変更しない）

### 1. 絵文字の最小化（目標: -5,000トークン/回、-5.5%）
**現状:**
```
:bar_chart: 日報 2026-02-24（月）
:clipboard: 主要アウトプット
:red_circle: 重大な意思決定・リスク対応
:large_orange_diamond: 業務判断・運用改善
:white_circle: 調整・連絡
:clock3: 時間配分
:bulb: 振り返り
:warning: ヒヤリハット
:thought_balloon: プロセス改善
:muscle: Good
:dart: 明日やること
:file_folder: 詳細ログ
```

**最適化:**
```
■ 日報 2026-02-24（月）

主要アウトプット
● 重大な意思決定・リスク対応
○ 業務判断・運用改善
・調整・連絡

時間配分
振り返り
! ヒヤリハット
+ プロセス改善
✓ Good

明日やること（事業インパクト順）
詳細ログ
```

絵文字 → シンプルな記号（■●○・!+✓）に変更。トークン消費を約5-10%削減。

### 2. プログレスバーの簡略化（目標: -2,000トークン/回、-2.2%）
**現状:**
```
レビュー/承認  ████████░░ 4.0h (38%)
意思決定      ██████░░░░ 3.0h (29%)
```

**最適化:**
```
レビュー/承認 4.0h (38%)
意思決定 3.0h (29%)
```

プログレスバー削除。数値のみ表示。

### 3. フォーマットの簡略化（目標: -5,000トークン/回、-5.5%）
**現状:**
- 冗長な説明、重複表現
- 「〜を確認し、〜を検討し、〜を実施」

**最適化:**
- 簡潔な表現に変更
- 箇条書きの階層を1階層に制限
- 「背景:」「補足:」などの冗長なラベルを削減

### 4. モデルの切り替え（目標: -40,000トークン/回、-44%）
**Haiku使用時の品質担保:**

データ収集は同じ（50件 × 複数クエリ）だが、分析・要約をHaikuで実行。

**品質維持のための工夫:**
- **明確な指示**: 分析基準を具体的に指定（「重要度=高は金額1000万以上 or 処理期限当日」など）
- **テンプレート化**: 出力フォーマットを厳密に指定
- **段階的処理**:
  1. データ分類（Haiku）
  2. 重要度判定（Haiku）
  3. 要約生成（Haiku）

Haikuでも質を保つため、指示を具体化：
```
以下の基準で分類せよ:
- 重大な意思決定: 金額1000万以上 OR 処理期限当日 OR 法務判断
- 業務判断: 金額100万以上 OR プロセス変更 OR 体制変更
- 調整・連絡: 上記以外
```

### 5. 詳細ログの最適化（目標: -10,000トークン/回、-11%）
**現状:**
- 全チャンネルの全やり取りを記載
- 重複表現が多い

**最適化:**
- カテゴリごとに最大5件まで
- 1件あたり最大100文字
- 定型フレーズを省略（「〜を確認」「〜を依頼」→「確認」「依頼」）

### 6. キャッシュ・重複排除（目標: -3,000トークン/回、-3.3%）
- 同じNotionページを複数回 `notion-fetch` しない
- Slackスレッドの重複取得を防ぐ
- ユーザー情報の重複取得を防ぐ

### 7. 出力の段階的制限（目標: -5,000トークン/回、-5.5%）
Subagentに指示：
```
出力トークン制限: 最大30,000トークン
この制限内で収めるため、以下の優先順位で削減:
1. 詳細ログの件数削減（各カテゴリ最大3件）
2. 振り返りの簡略化（各1-2文）
3. 明日やることの削減（最大5件）
```

## 品質モニタリング
Haiku使用時の品質チェック項目:
- [ ] 重要案件の漏れがないか（金額1000万以上、期限当日）
- [ ] 意思決定の理由が明確か
- [ ] 振り返りが具体的か（「忙しかった」等の抽象表現なし）
- [ ] 明日やることの優先度が妥当か

問題があれば、指示を具体化して再実行。
