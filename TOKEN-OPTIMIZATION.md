# トークン最適化設計

## 目標
月間トークン消費量を **3〜4.5％**（60〜90万トークン）に抑える。

## 現状分析（2026-02-25 実測値）
- 1回あたりのトークン消費: 約90,000トークン
  - Subagent（Sonnet）: 78,816トークン
  - メイン処理: 約11,000トークン
- 月30回実行: 約270万トークン（**13.5％**）

## 最適化施策

### 1. Slack検索の制限（目標: -20,000トークン/回）
**現状:**
- `conversations_search_messages`: limit=50（送信）+ limit=50（メンション）
- スレッド取得: 上位10件

**最適化:**
```markdown
### 1-A. Slack: 今日ユーザーが送信したメッセージ（DM含む）
conversations_search_messages: filter_users_from=@__SLACK_HANDLE__, filter_date_during=Today, limit=25
```

### 1-D. Slack: メンション（直近3日・未返信チェック兼用）
conversations_search_messages: search_query=@__SLACK_HANDLE__, filter_date_after=<3日前の日付>, limit=25
```

### スレッド文脈取得の制限
Step 2で上位10件 → **上位5件**に削減。

### 2. Notionページ取得の最適化（目標: -15,000トークン/回）
**現状:**
- 当日作成・更新ページを全件取得
- 各ページを `notion-fetch` で全文読み込み

**最適化:**
- 議事録以外のNotionページ: 最大5件に制限
- `notion-fetch` の読み込み深度: 第1階層のみ（子ブロックは取得しない）

### 3. SQLiteベースライン比較のオプション化（目標: -5,000トークン/回）
**現状:**
- 過去14日のベースライン比較を毎回実行

**最適化:**
- Slack Crawler DBパスが設定されている場合のみ実行
- 設定がない場合は「先週平均比」セクションをスキップ

### 4. 詳細ログの制限（目標: -10,000トークン/回）
**現状:**
- 全アクティビティを詳細ログに記載

**最適化:**
- 詳細ログは重要度「中」以上のみ記載
- ルーティン業務は件数のみ（詳細は省略）
- DM: 最大3件まで

### 5. Codex/Claude Codeログの要約（目標: -5,000トークン/回）
**現状:**
- セッションごとに全プロンプトを記載

**最適化:**
- セッション数のみカウント、詳細は最大3件まで
- プロンプトは最初の100文字に制限

## 最適化後の想定トークン消費
- Subagent: 約35,000トークン（78,816 → 35,000、-55%削減）
- メイン処理: 約10,000トークン
- **合計: 約45,000トークン/回**
- **月30回: 約135万トークン（6.75％）**

## さらなる最適化（目標達成のため）
上記施策で6.75％まで削減。3〜4.5％を達成するには追加施策が必要：

### 6. Subagent modelの変更
- Sonnet → **Haiku** に変更
- トークン消費: 約35,000 → **約20,000トークン**（-43%削減）
- 品質トレードオフ: 分析精度がやや低下する可能性

### 7. 隔日実行
- 毎日 → 隔日（月15回）に変更
- 月間トークン: 135万 → **67.5万トークン（3.4％）**

## 実装優先度
1. **必須（Phase 1）**: 施策1〜5を実装 → 6.75％まで削減
2. **オプション（Phase 2）**: 施策6または7を実装 → 3〜4.5％達成
